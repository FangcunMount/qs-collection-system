# Qlume 小程序 UI 设计规范

> 适用产品：Qlume 小程序
> 适用范围：总首页、医学量表、人格探索、行为能力，以及这些入口下的列表、详情、测评、结果与个人中心页面  
> 文档定位：仓库级设计真源（Design Source of Truth）  
> 建议路径：`/UI-DESIGN.md` 或 `/docs/design/UI-DESIGN.md`

---

## 0. 文档目的与使用方式

这份规范用于建立 Qlume 长期、统一、可复用的产品视觉语言。它既是设计与研发协作的依据，也是交给 AI 生成页面、组件或视觉资产时的固定上下文。

Qlume 的界面应让用户首先感到：**可信、平静、清楚、被照顾**。清新科技感和轻 3D 视觉用于增强品牌识别，不应削弱医疗信息的严肃性、测评流程的可理解性或儿童相关内容的安全感。

本规范的优先级如下：

1. 用户安全、隐私、可理解性与任务完成率；
2. 信息层级、状态准确性与交互一致性；
3. Qlume 品牌识别与四类入口的语义差异；
4. 视觉精致度、插图、动效与装饰效果。

若局部设计与上述优先级冲突，应优先保证前项。任何新增页面都应复用本文 Token 和组件规则，不得另起一套近似但不兼容的颜色、间距、圆角或阴影体系。

### 0.1 适用角色

- 产品：定义页面目标、用户任务、风险级别和内容优先级；
- 设计：使用本规范建立页面结构、视觉与交互状态；
- 前端：以语义 Token 和公共组件实现，不复制临时样式；
- 内容与医学团队：审查术语、风险提示、结果解释与行动建议；
- AI 编程助手：先读取本文，再根据具体页面任务输出设计判断和代码。

### 0.2 规则关键词

- **必须**：涉及一致性、安全性或可访问性的硬性要求；
- **应该**：默认遵循，只有明确理由时才可偏离；
- **可以**：按内容与场景选用；
- **禁止**：不得作为默认方案，除非规范明确说明例外。

### 0.3 修改原则

修改本规范时，应同步说明：变更原因、影响页面、Token 或组件迁移方式，以及旧实现的兼容期限。页面代码中的临时特例不能反向成为规范；只有经过复用验证的模式才应进入本文。

## 1. 产品语境与设计判断

### 1.1 产品定位

Qlume / 启明是一款面向家庭、儿童及相关照护者的互联网医疗与心理测评小程序。产品包含四层入口：

1. **总首页**：建立品牌认知，解释产品能力，帮助用户选择正确入口；
2. **医学量表**：提供专业、严谨、可追溯的筛查与量表测评；
3. **人格探索**：提供具有亲和力、启发性和自我认识价值的探索体验；
4. **行为能力**：围绕儿童日常行为、成长能力与可行动建议提供评估支持。

### 1.2 核心设计判断

> 这是一个面向儿童、家长与照护者的互联网医疗及心理测评产品。页面的首要目标是帮助用户在低压力环境中理解信息、选择合适的测评并安全完成任务。整体采用“温和的临床可信感”作为基础，以淡紫、蓝紫和青绿构成清新科技气质，以轻 3D 图标和插图建立识别度；信息结构保持理性克制，情绪表达保持关怀而不幼稚。

### 1.3 用户体验目标

所有页面都应支持以下感受：

- **我知道这里能做什么**：入口与行动明确；
- **我理解接下来会发生什么**：流程、耗时、适用对象和数据用途清楚；
- **我可以放心继续**：专业来源、隐私与风险说明可见但不制造恐慌；
- **我不会因为结果被贴标签**：使用解释性、非评判性语言；
- **我知道下一步怎么办**：结果页提供可执行建议及必要的专业求助路径。

### 1.4 设计目标与非目标

设计目标：

- 形成一眼可辨认但不过度装饰的 Qlume 视觉语言；
- 用统一产品骨架承载四类入口，用内容语义形成差异；
- 降低医学与心理测评的理解成本和完成压力；
- 让透明 PNG 图标和首页 Banner 成为受控品牌资产，而不是散落装饰；
- 让 AI 生成的页面也能保持长期一致性。

非目标：

- 不追求社交媒体式的强刺激、强转化或无限信息流；
- 不把医疗测评包装成游戏、运势、诊断结论或人格标签；
- 不为了“高级感”牺牲字号、对比度、信息密度和操作确定性；
- 不让四类入口发展为四套互不兼容的产品。

## 2. 设计原则

### 2.1 专业可信，证据先于装饰

- 测评名称、适用对象、预计时长、来源或专业说明应优先呈现；
- 风险、免责声明和隐私信息必须清楚，但避免用高饱和红色制造恐慌；
- 结果应区分“筛查/评估提示”和“医学诊断”，不得模糊表述；
- 装饰元素不得遮挡题目、选项、进度、结果或行动建议。

### 2.2 温和治愈，不幼态化

- 通过留白、圆润但克制的形状、柔和色彩和支持性文案减轻压力；
- 儿童场景可以更有亲和力，但不使用婴幼儿化字体、夸张卡通或奖励诱导；
- 避免“异常”“失败”“你有问题”等评判性语言；
- 使用“可能需要关注”“建议进一步了解”“可考虑咨询专业人员”等分级表达。

### 2.3 清晰优先，一屏一个主任务

- 每个页面必须有一个主要任务和一个主按钮；
- 主要内容先于品牌装饰，关键操作先于辅助入口；
- 页面结构应靠排版、间距和内容顺序建立，不依赖堆叠边框与阴影；
- 同级内容数量过多时，应分组、筛选或渐进披露，而不是缩小字号。

### 2.4 一个系统，四种语义

- 四类首页共用基础 Token、导航、卡片、按钮和状态组件；
- 差异来自内容主题、局部强调色、代表性插图与信息架构，不来自完全不同的组件皮肤；
- 用户跨入口时仍应感到处于同一个 Qlume 产品中。

### 2.5 一个视觉亮点，其余克制

- 每个页面最多设置一个主要视觉焦点；
- 首页视觉焦点通常是 Banner 或主题插图；
- 测评页视觉焦点必须是当前问题；
- 结果页视觉焦点应是结论摘要或关键趋势，而不是庆祝动画；
- 同屏不同时使用大渐变、光晕、玻璃拟态、悬浮 3D 和复杂动效。

### 2.6 状态是设计的一部分

正常、加载、空、错误、弱网、禁用、已完成、过期、无权限和风险提示必须在设计阶段覆盖。不得只设计理想数据状态。

### 2.7 尊重用户，不用焦虑驱动转化

- 不使用虚假倒计时、红点滥用、错过恐惧或结果悬念诱导；
- 不以“解锁完整结果”为由隐藏必要的安全信息；
- 不将分享、订阅或购买设置为查看核心结果的前置条件；
- 儿童相关信息默认遵守最小披露原则。

## 3. 品牌与视觉语言

### 3.1 品牌关键词

核心关键词：

- 专业可信
- 温和治愈
- 清新科技
- 理性清楚
- 陪伴成长
- 轻盈而不轻浮

反向关键词：

- 冰冷临床
- 过度童趣
- 神秘玄学
- 炫技科技
- 奢华医美
- 廉价促销

### 3.2 视觉母题：光、路径与生长

“Qlume / 启明”可使用“柔和的光”“可理解的路径”“逐步生长”作为长期视觉母题：

- 光：代表发现、理解和希望，使用局部柔和高光，不使用大面积霓虹；
- 路径：代表测评过程与行动建议，可用于进度、步骤与结果解释；
- 生长：代表儿童发展和自我认识，可使用芽、叶、轨迹、积木或抽象成长形态。

同一页面只应选用一个母题。母题必须参与信息表达，不能仅作为无意义背景纹理。

### 3.3 视觉比例

建议页面视觉构成为：

- 70% 中性背景、文字与结构；
- 20% 品牌浅色面、插图或分区色；
- 10% 主色、状态色和重点操作。

这不是像素级硬性计算，而是防止整页被蓝紫色和装饰填满的控制原则。

### 3.4 轻 3D 的定义

Qlume 的轻 3D 应具有：

- 柔和漫反射、低对比材质、轻微体积感；
- 统一视角，建议正视或轻微俯视，不混用多个透视角度；
- 柔和环境光和单一主光源；
- 低饱和、有限高光，无金属镜面和强烈霓虹；
- 简洁轮廓，缩小后仍可识别；
- 透明背景 PNG 或受控 WebP，边缘干净，无白边和杂色。

轻 3D 适用于入口图标、主题插图、空状态和少量结果解释；不适用于题目选项、表格数据、警告信息或每张卡片的重复装饰。

## 4. 色彩系统与语义 Token

### 4.1 色彩原则

- 代码只使用语义 Token，不直接使用“紫色 500”表达业务含义；
- 主色用于主操作、选中与品牌识别，不用于大段正文；
- 四类入口可以使用局部主题色，但基础文字、背景、边框和状态色保持统一；
- 文字和交互对比度必须满足第 16 节的可访问性要求；
- 渐变只允许用于 Banner、品牌插图或极小范围高光，不作为普通卡片默认背景。

### 4.2 核心色板

以下为建议初始值。正式接入已有品牌资产时，可以微调色值，但不得改变语义层级。

| 色彩角色 | Token | 色值 | 用途 |
|---|---|---:|---|
| 品牌蓝紫 | `brand-primary` | `#6657D9` | 主按钮、选中、关键链接、品牌识别 |
| 品牌蓝紫深 | `brand-primary-strong` | `#5144B8` | 按压态、强调文字、深色小面积图形 |
| 品牌淡紫 | `brand-lilac` | `#EEEAFE` | 浅色容器、标签、主题背景 |
| 清新青绿 | `brand-mint` | `#39BFA8` | 行为成长、正向进展、辅助强调 |
| 柔和天青 | `brand-sky` | `#5B9EEA` | 医学信息、路径与数据辅助色 |
| 暖光色 | `brand-warm` | `#F3B56A` | 少量温暖提示、插图高光，禁止大面积使用 |

### 4.3 中性色

| Token | 色值 | 用途 |
|---|---:|---|
| `neutral-0` | `#FFFFFF` | 浮层、主要卡片、反色文字背景 |
| `neutral-25` | `#FCFCFE` | 页面纯净背景 |
| `neutral-50` | `#F7F8FC` | 页面默认背景、分组背景 |
| `neutral-100` | `#EFF1F6` | 禁用背景、分隔区域 |
| `neutral-200` | `#E1E5ED` | 默认边框、分隔线 |
| `neutral-400` | `#99A1B2` | 占位符、弱辅助信息 |
| `neutral-600` | `#5D6678` | 次要正文、说明 |
| `neutral-800` | `#2D3340` | 标题与主要正文 |
| `neutral-950` | `#171A21` | 极强文字，仅小面积使用 |

### 4.4 状态色

| 语义 | 前景 Token / 色值 | 浅背景 Token / 色值 | 使用说明 |
|---|---|---|---|
| 成功 | `success` / `#218A70` | `success-soft` / `#E7F7F2` | 保存成功、完成、正常进展 |
| 提示 | `info` / `#347FC4` | `info-soft` / `#EAF4FD` | 普通信息、方法说明 |
| 注意 | `warning` / `#A96616` | `warning-soft` / `#FFF4E2` | 需要关注、过程提醒、非紧急风险 |
| 错误 | `error` / `#C64A55` | `error-soft` / `#FDECEF` | 输入错误、失败、明确高风险提示 |

状态不能只依靠颜色表达，必须同时使用文字、图标或结构提示。

### 4.6 四类入口主题映射

| 入口 | 主语义 | 局部强调色 | 浅色面 | 视觉意象 |
|---|---|---|---|---|
| 总首页 | 连接与发现 | `brand-primary` | `brand-lilac` | 柔光、入口汇聚、启明路径 |
| 医学量表 | 专业与清晰 | `brand-sky` | `info-soft` | 结构化路径、标尺、清晰光束 |
| 人格探索 | 理解与多样 | `brand-primary` + 少量 `brand-warm` | `brand-lilac` | 多面体、星图、层叠色块 |
| 行为能力 | 成长与行动 | `brand-mint` | `success-soft` | 生长轨迹、积木、叶芽、阶梯 |

主题色只影响 Banner、入口图标、局部标签和数据强调。主按钮默认仍使用品牌蓝紫；只有独立、连续的行为能力流程可以在经过一致性验证后使用青绿主操作。

### 4.7 渐变规则

允许的品牌渐变：

```css
--ql-gradient-brand-soft: linear-gradient(135deg, #F2EEFF 0%, #EAF5FF 58%, #E8F8F4 100%);
--ql-gradient-brand-focus: linear-gradient(135deg, #7568E5 0%, #5B9EEA 100%);
```

- `brand-soft` 仅用于总首页 Banner、空状态背景或大面积低对比插图底；
- `brand-focus` 仅用于品牌主视觉或特殊活动主按钮，常规按钮优先纯色；
- 禁止为每张卡片生成不同渐变；
- 渐变上叠加文字时必须单独验证最弱位置的对比度。

## 5. 字体与排版

### 5.1 字体栈

小程序默认使用系统字体，保证加载速度和中文可读性：

```css
font-family: -apple-system, BlinkMacSystemFont, "PingFang SC",
  "Microsoft YaHei", "Noto Sans CJK SC", sans-serif;
```

禁止仅为视觉差异引入体积较大的在线字体。品牌展示字体若确有需要，应只用于 Logo 或预渲染品牌图形，不用于正文和题目。

### 5.2 字体等级

| 角色 | 建议字号 | 行高 | 字重 | 用途 |
|---|---:|---:|---:|---|
| Display | `28px` | `38px` | `600` | 首页主标题、结果核心摘要 |
| H1 | `24px` | `34px` | `600` | 页面标题 |
| H2 | `20px` | `29px` | `600` | 主要区块标题 |
| H3 | `17px` | `25px` | `600` | 卡片标题、题目组标题 |
| Body L | `17px` | `27px` | `400` | 测评题目、重要说明 |
| Body | `16px` | `25px` | `400` | 默认正文、列表主文本 |
| Body S | `14px` | `22px` | `400` | 辅助说明、元信息 |
| Caption | `12px` | `18px` | `400/500` | 标签、时间、简短注释 |

### 5.3 排版规则

- 小程序正文不得默认小于 `14px`；题目和关键解释不得小于 `16px`；
- 中文正文通常不增加字间距，Display 可使用 `0` 至 `0.02em`；
- 单段正文建议控制在 3～6 行，长内容拆分为小标题、列表或折叠说明；
- 标题不使用全大写英文制造层级；
- 数字结果可使用等宽数字特性，但不得使用装饰字体；
- 文本默认左对齐。只有短标题、空状态和单一确认结果可以居中；
- 结果解读与医学说明的理想阅读宽度不超过约 32 个中文字符；
- 不使用纯颜色区分标题层级，字号、字重、间距必须共同承担层级。

### 5.4 文案语气

使用清楚、温和、非评判性的用户语言：

| 避免 | 推荐 |
|---|---|
| 你的孩子存在异常 | 结果提示有些方面值得进一步关注 |
| 测试失败 | 暂时未能提交，请检查网络后重试 |
| 必须立即就医 | 如出现以下情况，建议尽快联系专业人员 |
| 解锁你的隐藏人格 | 了解你在不同情境中的倾向 |
| 正常人群 | 参考人群 / 常模人群 |

涉及诊断、治疗、危机干预和量表效度的文案必须经过专业审核，AI 生成内容不得直接上线。

## 6. 间距、尺寸与网格

### 6.1 基础间距

使用 4px 基础单位，以 8px 为主要节奏：

| Token | 值 | 常见用途 |
|---|---:|---|
| `space-0` | `0` | 无间距 |
| `space-1` | `4px` | 图标内部、小元素微调 |
| `space-2` | `8px` | 标签间距、紧凑元素 |
| `space-3` | `12px` | 图标与文字、紧凑卡片 |
| `space-4` | `16px` | 默认组件内距、列表间距 |
| `space-5` | `20px` | 卡片内距 |
| `space-6` | `24px` | 页面水平边距、组件组间距 |
| `space-8` | `32px` | 区块间距 |
| `space-10` | `40px` | 大区块间距 |
| `space-12` | `48px` | 页面章节分隔 |
| `space-16` | `64px` | 首页大区块间距，谨慎使用 |

### 6.2 页面网格

- 以 `375px` 设计宽度为主要基准，同时适配 `320px` 至大屏手机；
- 默认页面水平安全边距为 `20px`，内容密集页可用 `16px`，不得低于 `16px`；
- 卡片默认内边距 `16px` 或 `20px`，同一页面保持一致；
- 首页大区块垂直间距 `32px`，表单和流程页区块间距 `24px`；
- 底部固定操作区必须包含设备安全区，并为正文预留等高底部空间；
- 不用负边距掩盖结构问题；Banner 与卡片轻微叠合仅允许作为统一首页模式。

### 6.3 触控尺寸

- 所有主要触控目标最小 `44 × 44px`；
- 相邻独立触控目标间的可感知间距不小于 `8px`；
- 单选题整行可点击，不要求用户只点小圆点；
- 删除、退出、中断测评等高风险操作不得紧贴主按钮。

## 7. 圆角、边框与阴影

### 7.1 圆角

| Token | 值 | 用途 |
|---|---:|---|
| `radius-xs` | `4px` | 小标签、进度条内部 |
| `radius-sm` | `8px` | 输入框、小型信息块 |
| `radius-md` | `12px` | 按钮、选项、普通卡片 |
| `radius-lg` | `16px` | 首页卡片、结果区块、弹窗 |
| `radius-xl` | `24px` | Banner、主题视觉容器 |
| `radius-full` | `999px` | 头像、状态点、短标签；禁止滥用 |

同一组件只能对应一个主圆角 Token。禁止为了“柔和”把所有容器都做成胶囊。

### 7.2 边框

- 普通边框：`1px solid var(--ql-color-border)`；
- 弱分隔：`1px solid var(--ql-color-border-subtle)`；
- 选中态：品牌色边框 + 浅品牌底，不只加阴影；
- 错误态：错误色边框 + 明确错误文案；
- 列表优先使用留白或分隔线，不为每一行加完整卡片边框。

### 7.3 阴影

```css
--ql-shadow-card: 0 4px 16px rgba(44, 51, 75, 0.06);
--ql-shadow-floating: 0 10px 28px rgba(44, 51, 75, 0.12);
--ql-shadow-focus: 0 0 0 3px rgba(102, 87, 217, 0.18);
```

- 普通页面优先用背景层级和边框，阴影不是卡片必需项；
- `shadow-card` 仅用于需要从背景抬起的重要卡片；
- `shadow-floating` 仅用于弹窗、浮层、底部抽屉；
- 同屏阴影层级不超过两级；
- 禁止彩色发光、黑色重阴影和所有卡片统一悬浮。

## 8. 图标与透明 PNG 资产

### 8.1 图标系统

- 功能图标必须来自同一线性图标家族；
- 建议线宽视觉等效 `1.75～2px`，圆角端点，轮廓简洁；
- 默认尺寸：行内 `16px`，常规 `20px`，导航 `24px`，入口图标 `40～56px`；
- 功能图标与装饰插图必须区分：前者表达操作，后者表达主题；
- 禁止使用 Emoji 代替正式图标；
- 图标必须配合可理解的文字，不能要求用户猜测抽象含义；
- 删除、警告、医疗风险等图标不使用可爱化 3D 表现。

### 8.2 透明 PNG 使用规则

现有大量透明 PNG 是品牌资产，应建立资产清单，至少记录：

- 文件名与语义名称；
- 所属入口与适用页面；
- 原始尺寸、推荐显示尺寸和像素密度；
- 主色、视角、光源和风格标签；
- 是否允许裁切、镜像、着色或叠加文字；
- 替代文本或无障碍说明；
- 当前版本与来源。

PNG 使用要求：

- 不拉伸、不改变宽高比，不重复压缩有损源文件；
- 默认完整显示，裁切必须确保主体和阴影不被截断；
- 透明边距过大时应在资产侧清理，不在各页面用不同负边距修正；
- 同屏最多使用一个主 3D 插图和 3～5 个入口级图标；
- 图片必须设置确定尺寸或比例，避免加载后布局跳动；
- 对体积较大的资产使用 WebP 等兼容格式并保留源文件；
- 非关键信息图片可延迟加载，首屏 Banner 应提供合适占位；
- 不能把重要文案烘焙进图片，必须保留可访问、可更新的文本层。

### 8.3 图标背景容器

入口图标可使用 `48～64px` 的浅色圆角方形底座。底座颜色来自对应主题浅色面，图标本体保持透明和清晰。禁止同时叠加高饱和渐变、粗边框、重阴影与光晕。

## 9. 插图与 Banner

### 9.1 插图风格

- 采用轻 3D 或柔和 2.5D，避免在同一流程混入扁平卡通、真实摄影和赛博 3D；
- 人物形象应具有包容性，不夸张表现疾病、情绪或儿童特征；
- 不使用白大褂、听诊器、红十字作为所有医学页面的默认符号；
- 不用大脑、心电图、星座等符号替代真实的信息解释；
- 插图承担入口识别、概念解释或情绪缓冲之一，不应三者混杂。

### 9.2 首页 Banner 规范

首页 Banner 是页面唯一主要视觉亮点，结构建议为：

```text
┌──────────────────────────────┐
│  短眉题 / 状态标签             │
│  一句清楚的主标题      主题插图 │
│  1～2 行支持性说明             │
│  [主要行动]  [次要文字行动]     │
└──────────────────────────────┘
```

- 标题应表达用户收益或当前任务，不使用空泛品牌口号；
- Banner 高度随内容自适应，移动端建议约 `180～240px`；
- 插图占比不应压缩标题和按钮的可用空间；
- 文字与插图可左右错位，但文字必须保持完整阅读区；
- 自动轮播默认关闭。确需多个 Banner 时，优先改为纵向内容区；
- 禁止把 Banner 做成广告轮播、满屏渐变 Hero 或纯装饰大图。

### 9.3 结果页插图

结果页插图只用于降低阅读压力或解释抽象概念。风险较高的结果不得使用庆祝彩带、奖杯、失败表情或夸张情绪人物。

## 10. 页面布局与通用结构

### 10.1 页面壳层

所有业务入口共享以下基础结构：

```text
系统状态栏 / 安全区
页面导航
页面标题或主题 Banner
主要任务区
辅助解释或推荐内容
必要的可信与隐私信息
底部操作区 / TabBar
设备安全区
```

### 10.2 导航规则

- 总首页使用品牌导航，可包含用户入口和消息入口；
- 二级首页显示明确入口名称，并提供返回总首页的可预测路径；
- 测评流程使用简化导航，只保留返回/退出、标题和必要帮助；
- 返回按钮行为必须符合微信小程序导航栈；
- 深链或分享进入时，必须提供回到该业务首页或总首页的路径；
- 页面标题不要同时出现在原生导航栏和内容区造成重复，除非两者承担不同层级。

### 10.3 内容层级

一个典型业务首页按以下优先级组织：

1. 当前身份、状态或推荐行动；
2. 页面主要目标和入口；
3. 最近任务、进度或结果；
4. 分类浏览与辅助内容；
5. 专业说明、隐私和帮助。

不要为了填满页面添加“热门推荐”“猜你喜欢”“每日知识”等与主任务无关的模块。

### 10.4 列表与卡片选择

- 同类连续内容优先使用列表；
- 需要独立操作、状态或明显分组时使用卡片；
- 不要把列表中的每一行都做成浮起的圆角卡片；
- 卡片内部信息顺序应稳定：标题 → 元信息 → 摘要/状态 → 操作；
- 卡片只允许一个主要点击行为，内部多个操作必须有清楚分区。

### 10.5 固定底部操作区

- 用于测评开始、下一题、提交、保存等关键操作；
- 背景为不透明或接近不透明的 `surface`，上方使用弱边框；
- 包含安全区内边距；
- 不遮挡错误提示、最后一项内容或键盘；
- 主次按钮不能同色同权重。

## 11. 四类首页规则

### 11.1 总首页

**唯一目标：** 帮助用户理解 Qlume 能提供什么，并进入最合适的业务入口。

推荐结构：

```text
┌─ 品牌导航 / 问候与个人入口 ───────┐
│  Qlume 主题 Banner               │
│  一句话价值 + 一个主要行动          │
└──────────────────────────────────┘

继续进行 / 最近结果（有数据时）

选择你想了解的方向
┌ 医学量表 ┐  ┌ 人格探索 ┐
└─────────┘  └─────────┘
┌ 行为能力 ┐  ┌ 其他受控入口 ┐
└─────────┘  └────────────┘

专业与隐私说明 / 帮助
```

规则：

- 三个核心入口不能仅靠颜色和抽象图标区分，必须有一行用户语言说明；
- 有未完成任务时，“继续进行”优先于新测评推荐；
- Banner 不承载超过两个操作；
- 不使用三张完全等宽、同权重的 SaaS 功能卡作为桌面式布局；移动端可用 2 列入口，但尺寸和内容权重应有区分；
- 总首页保持品牌统一，不同时铺满三种主题色。

### 11.2 医学量表首页

**唯一目标：** 帮助用户安全地找到、理解并开始合适的专业量表。

必须优先展示：

- 测评适用对象；
- 预计用时和题目数量；
- 填写者身份要求，如本人、家长或专业人员；
- 测评用途和非诊断声明；
- 已有任务状态和继续入口。

视觉规则：

- 以天青和中性色建立清晰、专业感，蓝紫仅用于品牌和主操作；
- 使用结构化列表、筛选和标签，不使用娱乐化卡牌；
- 高风险或专业限制信息放在内容上下文附近，不只藏在底部协议；
- 量表详情应先回答“是否适合我/孩子”，再介绍背景与来源。

推荐结构：

```text
专业、清晰的主题区
继续进行 / 待完成任务
按需求或适用对象选择量表
量表列表（对象、时长、用途、状态）
如何选择 / 专业说明 / 隐私说明
```

### 11.3 人格探索首页

**唯一目标：** 以轻松但不轻率的方式，引导用户开始有解释价值的人格探索。

视觉规则：

- 可以比医学量表更具情绪和图形表现，使用淡紫、多面体或层叠形态；
- 可少量加入暖光色，避免星座、占卜、水晶球等玄学暗示；
- 文案强调“倾向、情境、理解和探索”，不强调固定类型和命定标签；
- 分享卡片不得披露敏感结果，不通过夸张人格称号追求传播。

推荐结构：

```text
人格探索主题 Banner
继续探索 / 最近结果
推荐探索（突出一个，不并列堆三张）
按主题浏览：关系、情绪、偏好等
结果如何理解 / 隐私说明
```

### 11.4 行为能力首页

**唯一目标：** 帮助家长或照护者理解儿童行为与成长能力，并获得具体下一步建议。

视觉规则：

- 使用青绿、生长路径、积木或抽象能力模块；
- 清楚标示观察周期、填写者身份和适用年龄；
- 不把儿童能力展示为排名、输赢或通关等级；
- 结果重点是能力画像、环境差异与可行动建议，不是单一总分；
- 可展示成长记录，但不得暗示短期波动等于能力退步。

推荐结构：

```text
成长与行动主题区
需要继续的观察或评估
按年龄 / 场景选择能力方向
能力主题列表
家庭行动建议入口
专业支持与使用说明
```

### 11.5 四类首页的一致性要求

必须共用：

- 导航高度与返回逻辑；
- 页面边距、区块节奏、字体等级；
- 按钮、卡片、标签、加载和错误组件；
- 任务状态表达；
- 隐私和专业说明入口；
- 埋点命名与无障碍策略。

允许差异：

- 主题浅色面和局部强调色；
- Banner 插图与视觉母题；
- 内容分类方式；
- 解释语气的轻重程度；
- 与业务相符的标志性组件。

## 12. 组件规范

### 12.1 按钮

#### Primary

- 用于当前页面唯一主要行动；
- 默认高度 `48px`，圆角 `12px`，左右内距不小于 `20px`；
- 纯品牌蓝紫底、白字；
- 同屏通常只有一个 Primary；
- 必须包含 default、pressed、loading、disabled 状态。

#### Secondary

- 浅品牌底 + 深品牌文字，或白底 + 品牌边框；
- 用于不推进主流程的次要行动；
- 不得与 Primary 视觉权重相同。

#### Tertiary / Text

- 用于帮助、跳过、了解更多等低权重操作；
- 点击区域仍需达到 `44px` 高度或提供足够内距；
- 危险操作使用错误色文字，并经二次确认。

按钮文案使用明确动词，如“开始测评”“继续填写”“查看建议”，避免“确定”“好的”“立即解锁”等模糊或施压词语。

### 12.2 卡片

默认卡片：

- 背景 `surface`；
- 圆角 `radius-lg`；
- 内距 `16px` 或 `20px`；
- 默认使用弱边框，可按页面层级选择轻阴影；
- 标题、状态和操作对齐规则保持一致。

禁止在每张卡片顶部放无意义编号、彩色渐变条或装饰光球。可点击卡片必须有 pressed 反馈，并避免在卡片内嵌套多个模糊点击区域。

### 12.3 量表/测评列表项

至少包含：

- 量表或任务名称；
- 适用对象/年龄；
- 预计用时或题目数量；
- 状态，如未开始、进行中、已完成、已过期；
- 一个明确行动。

专业来源、版本和填写者限制可在详情页展示，不应把列表塞成小字说明墙。

### 12.4 标签与徽标

- 标签用于分类、状态或短属性，不用于完整句子；
- 高度建议 `24～28px`，圆角 `6～8px`；
- 同屏标签颜色不超过 3 种；
- 红点仅用于真实、需要注意的未读或异常状态；
- 禁止把所有元信息都做成胶囊。

### 12.5 表单与输入框

- 输入框高度不小于 `48px`，多行输入有明确最小高度；
- 标签置于输入框上方，不能只用 placeholder 充当标签；
- 必填、格式说明和错误信息靠近对应字段；
- 错误出现时不清空已输入内容；
- 键盘类型、最大长度和自动聚焦应与字段匹配；
- 身份、儿童信息和健康信息要说明用途与可见范围；
- 禁止默认收集完成任务不需要的信息。

### 12.6 单选、多选与题目选项

- 整个选项容器可点击，默认最小高度 `52px`；
- 选中态同时使用边框、背景、控件和必要文字提示；
- 选项文字允许自然换行，不截断医学或行为描述；
- 量表原始题序和选项顺序不得为视觉美观随意改动；
- 自动进入下一题只在不会引发误操作且经过验证时使用；
- 多选题必须明确“可多选”及数量限制。

### 12.7 进度指示

- 显示可理解的进度，如“第 6 / 20 题”，必要时配合进度条；
- 进度条使用品牌色或入口主题色，轨道使用中性浅色；
- 不显示虚假的完成百分比；
- 分支量表导致总题数变化时，优先显示步骤或阶段而不是精确百分比；
- 进度不得通过焦虑性文案施压。

### 12.8 导航栏与 TabBar

- 导航图标统一使用 `24px` 图标家族；
- TabBar 最多 5 项，标签使用用户语言；
- 选中态使用品牌色 + 图标形态或字重变化，不能只靠颜色；
- 业务二级首页不随意新增独立 TabBar；
- 自定义导航必须适配胶囊按钮位置和顶部安全区。

### 12.9 弹窗、抽屉与提示

- 短确认使用居中弹窗，复杂选择或说明使用底部抽屉/独立页面；
- 标题直接说明决定，如“要退出本次测评吗？”；
- 明确说明结果和是否保存进度；
- 取消操作通常保持低风险默认位置；
- 不连续弹出多个弹窗；
- Toast 只承载短暂反馈，不承载关键错误说明。

### 12.10 结果摘要与数据表达

- 先给出人能理解的摘要，再给出分数、维度和专业解释；
- 图表必须有文字摘要、单位、参考范围和数据来源说明；
- 不使用仪表盘把复杂心理结果简化成“好/坏”；
- 不用红绿两色作为唯一差异；
- 维度较多时优先使用条形图、小倍数图或分组列表，谨慎使用雷达图；
- 结果底部提供明确行动：保存、查看建议、咨询专业人员或返回首页。

### 12.11 可信与隐私说明组件

用紧凑、可展开的信息块说明：

- 测评用途与局限；
- 数据保存和使用方式；
- 谁可以查看；
- 何时建议寻求专业帮助；
- 联系方式与紧急情况指引。

高风险信息不能仅存在于折叠区或用户协议中。

## 13. 测评流程专项规范

### 13.1 开始前

开始页必须回答：

- 这是什么；
- 适合谁；
- 由谁填写；
- 大约需要多久；
- 需要回忆或观察哪个时间范围；
- 结果能说明什么、不能说明什么；
- 数据将如何使用；
- 是否支持中途保存。

主操作为“开始测评”。协议确认不得使用默认勾选。

### 13.2 进行中

- 题目是唯一视觉中心；
- 保留进度、必要帮助和退出路径；
- 装饰插图应极少或完全移除；
- 题目和选项原文保持完整；
- 页面切换不能丢失已选答案；
- 弱网、保存中和保存失败必须有可恢复反馈；
- 长量表应支持经业务允许的断点继续。

### 13.3 提交前

- 提示未答题数量及返回检查方式；
- 明确提交后是否可修改；
- 防止重复提交，提交中按钮进入 loading 并锁定；
- 网络失败时保留本地答案并提供重试；
- 不用庆祝动画暗示某种结果。

### 13.4 结果页

建议顺序：

1. 结果摘要；
2. 适用范围和局限；
3. 关键维度或观察点；
4. 具体建议；
5. 必要的专业求助提示；
6. 查看详情、保存或返回行动。

结果不应把用户或儿童固化为一个标签。严重风险提示必须使用清楚、平静、可行动的文案，并提供经过业务确认的联系路径。

## 14. 状态规范

### 14.1 Loading

- 首屏骨架应匹配真实结构，避免整页转圈；
- 操作提交使用按钮内 loading，并阻止重复操作；
- 超过约 2 秒可显示简短解释；
- 不伪造进度；
- 图片加载失败时保留布局和替代内容。

### 14.2 Empty

空状态包含：发生了什么、为什么可能为空、用户下一步可以做什么。

示例：

- “还没有进行中的测评” + “浏览可用量表”；
- “暂时没有成长记录” + “完成首次评估后在这里查看”；
- 搜索无结果时保留搜索词并提供清除筛选。

插图应轻量，不能比行动按钮更突出。

### 14.3 Error

错误信息需区分：

- 输入错误：字段附近说明并保留输入；
- 网络错误：说明可重试，保留当前数据；
- 服务错误：提供重试和稍后再试路径；
- 权限错误：说明需要什么权限及用途；
- 业务不可用：说明原因、影响和替代路径；
- 高风险异常：提供明确联系或求助方式。

禁止只显示“请求失败”“未知错误”或内部错误码。错误码可作为辅助信息复制，但不能替代用户解释。

### 14.4 Offline / Weak Network

- 能本地保存的测评答案优先保存；
- 顶部或题目区显示非阻断性弱网提示；
- 恢复网络后自动同步需明确反馈；
- 冲突时不得静默覆盖，需按业务规则提示；
- 离开页面前明确说明未同步状态。

### 14.5 Disabled

- 禁用状态必须有原因；
- 不应让用户点击后才知道条件不满足；
- 若条件可完成，应在附近说明完成路径；
- 禁用颜色仍需保证文本可读。

### 14.6 Success

- 使用简短、具体反馈，如“已保存”“已提交”；
- 重要成功状态提供下一步；
- 不对敏感测评结果使用夸张庆祝；
- Toast 消失后，页面本身仍应反映成功状态。

### 14.7 权限与隐私状态

- 权限请求必须由用户动作触发，并先说明用途；
- 用户拒绝后提供不授权的可用路径或设置指引；
- 不反复弹窗催促；
- 分享前预览将要披露的内容，默认隐藏敏感数据。

## 15. 动效与触觉反馈

### 15.1 动效原则

动效只能服务以下目的之一：

- 表达信息层级；
- 解释状态变化；
- 确认用户操作结果；
- 帮助理解页面或流程关系。

### 15.2 时长与曲线

| 场景 | 时长 | 建议曲线 |
|---|---:|---|
| 按压与微反馈 | `100～160ms` | `ease-out` |
| 普通状态切换 | `160～240ms` | `ease-out` |
| 抽屉、弹窗 | `220～320ms` | 标准减速曲线 |
| 页面级过渡 | `250～400ms` | 与原生导航协调 |

- 优先使用 `opacity` 和 `transform`；
- 避免频繁改变布局属性导致跳动；
- 选项点击可有轻微背景与边框过渡，不让整卡持续漂浮；
- 结果图表可进行一次性、短暂呈现，不循环播放；
- 尊重系统减少动态效果设置，必要时提供无动画等效状态。

### 15.3 禁止动效

- 无意义循环呼吸、漂浮、旋转和发光；
- 大量卡片依次飞入；
- 影响答题速度的题目切换动画；
- 对风险结果使用彩带、烟花、震动或失败抖动；
- 通过动效隐藏等待时间或阻止用户操作。

---

## 16. 可访问性与包容性

### 16.1 对比度

- 普通文字与背景对比度至少 `4.5:1`；
- 大字号或粗体大文字至少 `3:1`；
- 关键图标、边界和交互状态至少 `3:1`；
- 渐变和图片上的文字要在最弱背景位置验证；
- `text-muted` 只用于非关键信息。

### 16.2 非颜色表达

- 成功、警告和错误同时使用图标/文字/结构；
- 图表使用形状、标签或纹理辅助颜色；
- 选中态同时改变控件、边框或标记；
- 不用红绿表达儿童能力的“好/坏”。

### 16.3 文字与操作

- 正文、题目和结果支持系统字体放大时不截断；
- 触控目标至少 `44 × 44px`；
- 焦点顺序与视觉顺序一致；
- 所有图片有适当替代信息；纯装饰图应被辅助技术忽略；
- 图标按钮必须有可读名称；
- 重要说明不用只在 hover 中出现，小程序触屏没有可靠 hover。

### 16.4 认知与情绪可访问性

- 每屏控制主要选择数量，复杂任务分步呈现；
- 使用短句、具体动词和稳定术语；
- 不用模糊倒计时和压力性文案；
- 允许用户返回检查和恢复未完成任务；
- 对敏感问题提供必要说明和退出路径；
- 结果解释避免标签化、羞耻化和确定性断言。

### 16.5 儿童与照护者场景

- 明确当前填写者和被评估对象，避免填错身份；
- 年龄、监护关系、隐私授权等信息在使用前解释；
- 不鼓励儿童独自处理需要监护人或专业人员参与的结果；
- 分享和截图场景默认最小化儿童身份信息。

## 17. 微信小程序实现约束

### 17.1 尺寸与适配

- 可用 `rpx` 适配布局，但字号、细边框和图标需在真机检查；
- 设计稿按 `375px / 750rpx` 建立映射时，不机械放大所有元素；
- 使用安全区变量处理刘海、底部指示条和横屏异常；
- 自定义导航栏必须读取系统信息和胶囊按钮位置；
- 固定底栏需要为 `safe-area-inset-bottom` 预留空间；
- 至少验证小屏、主流尺寸、大屏和系统大字体。

### 17.2 性能与包体

- 首页首屏只加载当前必要资源；
- 大型透明 PNG 应压缩并按实际显示尺寸输出多规格；
- 合适时使用 WebP，但必须验证目标微信版本兼容性和透明通道；
- 业务资源按医学量表、人格探索、行为能力分包，公共 Token 与基础组件保留在主包；
- 分包不得复制相同大图、字体或组件运行时代码；
- 避免在 WXML 中生成过深层级和超长列表；长列表使用分页、虚拟列表或按需渲染；
- 动画优先使用合成友好属性，避免低端设备卡顿。

### 17.3 组件与样式工程

- Token 应集中在公共样式或主题配置中，并提供 CSS 变量不可用场景的编译方案；
- 页面不得复制一份修改数值后的按钮、卡片和状态组件；
- 组件 API 使用业务无关的稳定语义，如 `variant="primary"`、`status="warning"`；
- 四类入口通过主题上下文或受控 class 覆盖局部语义色，不在组件内部判断业务名称；
- 原生组件层级、遮罩、滚动容器和键盘顶起行为必须在真机验证；
- 不依赖 hover 作为关键反馈，使用 `hover-class` 或明确 pressed 状态；
- 所有异步操作要防重复提交，并处理页面卸载后的回调。

### 17.4 图片与 CDN

- 明确图片宽高，使用合适 `mode`，禁止无意拉伸；
- 远程图片域名必须配置并使用 HTTPS；
- 首屏图提供本地或轻量占位；
- CDN 资源要有版本或内容哈希，避免旧 Banner 长期缓存；
- 失败时提供占位和重试，不显示破损图标；
- 敏感信息不能通过图片 URL 参数暴露。

### 17.5 微信能力

- 授权、登录、订阅消息和分享必须在明确用户动作后触发；
- 分享卡片标题与图片不得泄露具体测评结果、儿童姓名或风险状态；
- 需要手机号、头像等信息时说明用途，遵循最小采集；
- 使用原生隐私协议与授权能力，并保持页面说明一致；
- 分享、返回和重新进入时要恢复正确任务状态。

### 17.6 测试基线

每个关键页面至少验证：

- iOS 与 Android 真机；
- 小屏与主流屏幕；
- 浅色/系统相关设置；
- 大字体；
- 弱网、断网和接口超时；
- 首次授权、拒绝授权和已授权；
- 空数据、长文本、超长名称和异常图片；
- 测评中断、恢复、重复提交和分享进入。

## 18. 禁止项与常见 AI 模板味

除非产品语境和页面目标明确需要，禁止默认使用：

- 蓝紫色霓虹渐变铺满整个页面；
- 深色背景、发光网格和居中巨大 Hero；
- 三张完全等宽、同权重、同结构的功能卡；
- 所有容器玻璃拟态、背景模糊和半透明；
- 每张卡片一个彩色渐变、图标圆底和重阴影；
- 无意义的 `01 / 02 / 03` 编号；
- 大量胶囊按钮与胶囊标签；
- 右上角装饰光球、漂浮圆点、随机星星和网格纹理；
- 默认 Inter 风格、英文大标题和不符合中文语境的排版；
- 为了高级感随意使用衬线字体；
- 卡片 hover 上浮、持续呼吸、循环发光等无意义动画；
- Emoji 充当导航和功能图标；
- 使用库存人物照片制造“医疗可信感”；
- 把所有信息都放进卡片，导致页面没有层级；
- 为填满页面添加虚构统计数据、用户评价、合作机构或专业背书；
- 没有来源的“AI 智能诊断”“准确率 99%”等主张；
- 将医学筛查结果表达为确定诊断；
- 使用焦虑、羞耻、排名、通关或失败机制推动儿童测评；
- 隐藏加载、错误、空状态和隐私授权设计；
- 在代码中散落近似色值、魔法间距和一次性组件。

判断一个页面是否有 AI 模板感时，重点检查：如果替换 Logo 和文案，它是否会立刻变成任意 SaaS、AI 工具或心理测试网站？如果答案是“会”，说明设计尚未从 Qlume 的用户、任务、内容和已有资产中生长出来。

## 19. AI 生成页面的 Prompt 使用规则

### 19.1 固定上下文

任何 AI 设计或编码任务都必须提供：

1. 本 `UI-DESIGN.md`；
2. 当前页面的真实业务目标和目标用户；
3. 真实内容结构、字段、状态和接口约束；
4. 已有组件、Token 和资产清单；
5. 相关页面截图或当前实现；
6. 明确禁止修改的业务逻辑和兼容约束。

不得只输入“设计一个高端、清新、有科技感的页面”。这种描述会诱导模型使用通用渐变、玻璃卡片和装饰光晕。

### 19.2 每次任务必须填写的页面简报

```markdown
## 页面任务

- 页面名称：
- 所属入口：总首页 / 医学量表 / 人格探索 / 行为能力
- 页面唯一目标：
- 主要用户：
- 使用场景：
- 主要内容与真实字段：
- 主要操作：
- 次要操作：
- 风险级别与专业限制：
- 必须覆盖的状态：
- 已有组件：
- 可用图片与图标：
- 技术栈与页面路径：
- 明确不能改变的内容：
- 验收标准：
```

### 19.3 AI 输出顺序

AI 不得未经分析直接生成代码。输出顺序必须为：

1. 一句话设计判断；
2. 页面唯一目标、主要任务和信息优先级；
3. 复用的现有 Token、组件和资产；
4. 页面信息架构；
5. 移动端 ASCII 线框图；
6. 关键组件及全状态设计；
7. 与本规范的偏离项及理由；
8. 实现代码；
9. 自我审查与验证结果。

### 19.4 可直接复用的页面生成 Prompt

```markdown
你是 Qlume / 启明小程序的资深产品设计负责人和微信小程序前端工程师。

先完整阅读仓库中的 UI-DESIGN.md。它是本任务的设计真源；不得另建一套颜色、字体、间距、圆角、阴影、图标或状态体系。

Qlume 是互联网医疗与儿童心理测评小程序，品牌气质为专业可信、温和治愈、清新科技。淡紫、蓝紫、青绿和轻 3D 是受控品牌语言，不代表可以默认使用大面积渐变、玻璃拟态、光晕或悬浮卡片。

本次页面简报：
[粘贴填写后的页面简报]

已有代码与资产：
[粘贴相关组件路径、接口、字段、PNG/Banner 资产与截图]

请先检查当前仓库实现和可复用组件，再进行设计。不要虚构接口、数据、专业结论、机构背书、统计数字或用户评价。不要改变量表题目、选项顺序、评分含义和既有业务流程，除非任务明确授权。

设计要求：

- 一屏一个主要任务，一个页面通常只有一个主按钮；
- 从真实用户、任务、医学/心理内容和已有品牌资产中形成设计，不套用通用 AI SaaS 模板；
- 四类入口共享产品骨架，只通过受控主题色、插图和内容结构体现差异；
- 先用排版、间距和内容顺序建立层级，再考虑边框、阴影和装饰；
- 覆盖 loading、empty、error、offline、disabled、success、权限和长文本状态；
- 满足触控尺寸、颜色对比、非颜色表达、系统大字体和安全区要求；
- 适配微信小程序性能、分包、图片体积、原生组件层级和真机行为；
- 医学与儿童相关文案保持非诊断、非评判、可行动，标出需要专业审核的内容。

按以下顺序输出：

1. 一句话设计判断；
2. 信息优先级和页面结构；
3. 复用 Token、组件与资产清单；
4. 375px 宽度 ASCII 线框图；
5. 组件状态与交互说明；
6. 实现代码；
7. 对照 UI-DESIGN.md 的自我审查；
8. 已完成的构建、真机或状态验证，以及尚未验证的风险。
```

### 19.5 AI 修改现有页面规则

- 先阅读当前代码、公共样式、路由、组件和真实资产；
- 保留已有业务行为、数据契约、埋点、分享与授权逻辑；
- 优先复用和小步修改，不重写整个页面以追求视觉统一；
- 不替换已有真实 PNG/Banner 为临时占位图；
- 不在没有视觉证据时声称“完全还原”；
- 修改后必须列出影响页面和复用组件，并运行与风险相称的验证；
- 如果规范与现有实现冲突，先报告冲突与迁移方案，不静默扩大改动。

### 19.6 AI 视觉资产生成规则

生成新插图或图标前，Prompt 必须包含：

- 资产用途和显示尺寸；
- 所属入口和视觉母题；
- 已有资产的视角、材质、光源、色板与轮廓特征；
- 透明背景要求；
- 禁止出现的医疗误导、文字、Logo、人物特征或装饰；
- 缩小到目标尺寸后的识别要求。

生成结果必须经过人工检查：透明边缘、光源一致性、品牌色偏差、儿童与医疗表达、缩小可读性、版权与来源。

### 20.1 产品与内容

## 21. 落地与治理建议

### 21.1 推荐仓库结构

```text
docs/design/
├── UI-DESIGN.md             # 本规范：原则、系统与生成规则
├── TOKENS.md                # 可选：Token 与代码映射
├── COMPONENTS.md            # 可选：公共组件 API 与示例
├── ASSET-CATALOG.md         # PNG、Banner、插图资产清单
└── decisions/               # 重要设计决策记录

src/
├── styles/tokens.*          # 代码中的设计 Token
├── components/ui/           # 基础 UI 组件
└── assets/                  # 按公共与分包管理的资产
```

### 21.2 单一真源

- 本文负责原则、视觉语言、通用模式和审查门槛；
- 代码 Token 文件负责可执行色值和尺寸；
- 组件文档负责 API、状态和用例；
- 资产清单负责现有 PNG/Banner 的来源和使用边界；
- 业务 PRD 负责页面内容、流程和医学规则。

不同文档不得重复维护同一事实。若代码与本文不一致，应通过明确的设计决策更新一方，不长期保留隐性偏差。

### 21.3 采用步骤

1. 盘点现有色值、字体、间距、圆角、阴影和基础组件；
2. 为透明 PNG 与 Banner 建立资产清单；
3. 将现有样式映射到语义 Token，标记例外和迁移成本；
4. 先统一按钮、卡片、状态、导航和测评选项等高复用组件；
5. 选择总首页与一个测评流程作为基准页面；
6. 完成四类首页差异化验证；
7. 把本文加入 AI 助手上下文和 PR 设计审查模板；
8. 用真实设备、真实长文案和异常状态持续校准。

### 21.4 例外流程

偏离本规范时，应在设计稿或 PR 中记录：

- 偏离的规则；
- 用户或业务原因；
- 影响范围；
- 是否需要形成新 Token/组件；
- 是否是临时方案及清理时间。

单个页面的视觉偏好不能成为例外理由。只有能改善真实任务、可访问性、性能或业务准确性的证据，才应推动规范演进。

## 22. 最终验收标准

一份符合本规范的 Qlume 页面，应同时满足：

- 用户能在数秒内理解页面用途和主要行动；
- 医疗与儿童心理语境被专业、温和、非评判地表达；
- 页面明显属于 Qlume，但不依赖大面积蓝紫渐变证明品牌；
- 轻 3D、PNG 和 Banner 有明确功能与数量边界；
- 四类入口各有语义个性，又共享统一产品骨架；
- 正常与异常状态都能安全完成或恢复任务；
- 小屏、大字体、弱网、权限拒绝和真机环境可用；
- 代码使用语义 Token 和公共组件，没有继续制造视觉债务；
- 即使由 AI 生成，也能说明每个关键设计决策与 Qlume 产品语境的关系。

当页面只剩“好看”但无法回答为什么适合 Qlume、为什么适合这个用户、为什么有助于当前任务时，它还没有完成设计。
