# UI Baseline

本目录记录全链路前端重构的微信开发者工具截图基线。截图只用于评审，不参与小程序构建。

## Device Matrix

| Key | Viewport | Purpose |
| --- | --- | --- |
| `small` | 320 × 568 | 检查小屏、固定底栏和长文本拥挤 |
| `standard` | 375 × 812 | 主要视觉验收基线 |
| `large` | 430 × 932 | 检查大屏留白和卡片密度 |

## Scenario Matrix

| Area | Required states |
| --- | --- |
| 首页 | 已登录有报告、无报告、机构任务入口 |
| 目录 | 加载完成、空数据、长标题、分类筛选 |
| 作答 | 测评准备、单档案、多档案、单题、多题、校验失败 |
| 记录 | 首次加载、空数据、已出报告、处理中、失败、筛选面板 |
| 报告 | 医学报告、人格报告、缺少可选段落、等待、失败 |
| 我的 | 未登录、已登录、档案列表、表单校验错误 |

## Capture Rules

1. 使用 Node 16 构建最新 `dist`，再由微信开发者工具打开项目。
2. 每次只更新当前重构批次涉及的截图。
3. 文件名使用 `<area>--<state>--<device>.png`，例如 `records--empty--small.png`。
4. 截图前固定测试数据和系统字体大小；不得使用设计稿代替运行时截图。
5. 对比时重点检查导航胶囊避让、内容溢出、触控区域、固定底栏、安全区、颜色和字重。
6. 任何有意视觉变化都必须在批次说明中记录；无法解释的差异视为回归。

## 2026-07-14 Standard-screen Review

`review-2026-07-14/` 保存用户提供的 10 张 375×812 左右运行时截图，覆盖首页、医学目录/列表、人格目录/模型、行为能力、记录中心和医学报告。它们是本轮视觉纠偏的输入证据，不是纠偏后的最终基线。

本轮从截图识别并已在代码中修正的问题：

- 人格目录重复展示“深度探索测评”；
- 记录中心使用大面积领域色卡片和多层按钮，阅读密度偏低；
- 医学报告趋势与因子区块存在卡片嵌套；
- 因子图表只读取旧字段别名，且开发者工具中新 Canvas 路径可能出现空白；
- 订阅提醒悬浮按钮遮挡固定底部操作区。

随后一轮运行时反馈继续识别并修正了：首页最近报告标题与间距、医学分类/热门列表层级、测评准备页兜底图与主按钮对齐、多题问卷选项节奏与固定底栏遮挡，以及医学报告默认因子视图空白。默认因子视图现使用 DOM 概览，避免把不可渲染的 Canvas 空白当作视觉区域。

上述截图必须保留作为 before 证据。重新构建 production `dist` 后，已补拍人格目录和记录中心的 post-fix 标准屏，确认重复卡片消失且记录列表层级降低；其余 8 个场景仍需按相同数据补拍。关键页面还需补 `small` 与 `large`，不得用现有截图替代。
